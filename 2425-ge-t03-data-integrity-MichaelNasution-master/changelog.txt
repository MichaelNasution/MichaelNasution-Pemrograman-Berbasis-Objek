# T03 Data Integrity - Changelog

## Implementation Summary
Academic Simulator telah diimplementasikan dengan fokus pada data integrity, duplicate prevention, dan proper exception handling.

## Architecture Overview

### Core Components

#### Model Classes (Immutable Design)
**Course.java**
- Final fields: id, name, credits, grade
- Only provides getters (accessor-only)
- Immutable design ensures data protection
- toString() = "id|name|credits|grade"

**Student.java**
- Final fields: id, name, academicYear, major
- Only provides getters (accessor-only)
- Immutable design ensures data protection
- toString() = "id|name|academicYear|major"

**Enrollment.java**
- Final fields: courseId, studentId, academicYear, semester
- Mutable grade field for updating after creation
- toString() = "courseId|studentId|academicYear|semester|grade"

#### Exception Classes
- **CourseNotFoundException**: Thrown when course ID not found
- **StudentNotFoundException**: Thrown when student ID not found
- **DuplicateDataException**: Thrown when attempting to add duplicate data

#### Manager Classes (Singleton Pattern with Static Methods)

**CourseManager.java**
- Static TreeMap<String, Course> for sorted storage
- addCourse(Course): Throws DuplicateDataException if ID exists
- getCourse(String): Retrieves course or throws CourseNotFoundException
- courseExists(String): Boolean check without exceptions
- getAllCourses(): Returns TreeMap in sorted order

**StudentManager.java**
- Static TreeMap<String, Student> for sorted storage
- addStudent(Student): Throws DuplicateDataException if ID exists
- getStudent(String): Retrieves student or throws StudentNotFoundException
- studentExists(String): Boolean check without exceptions
- getAllStudents(): Returns TreeMap in sorted order

**EnrollmentManager.java**
- Static TreeMap<String, Enrollment> for sorted storage
- Composite key: courseId-studentId-academicYear-semester
- addEnrollment(String...): Validates course FIRST, then student per spec
- Throws CourseNotFoundException if course invalid
- Throws StudentNotFoundException if student invalid
- Throws DuplicateDataException if enrollment exists
- getAllEnrollments(): Returns TreeMap in sorted order

#### Driver Classes

**Driver1.java** (Basic Testing)
- Accepts: course-add, student-add, enrollment-add commands
- Input format: command#param1#param2#param3#param4
- Silently ignores duplicate attempts and validation errors
- Outputs all data in sorted order: courses, students, enrollments
- Usage: academic.driver.Driver1

**Driver2.java** (Exception Handling)
- Same input format as Driver1
- Displays error messages for invalid course/student:
  - Format: "invalid course|{courseId}"
  - Format: "invalid student|{studentId}"
- Continues execution after errors
- Outputs all data in sorted order after error messages
- Usage: academic.driver.Driver2

## Key Design Decisions

### 1. Immutability for Data Protection
- Course and Student are fully immutable (no setters)
- Rationale: Prevents accidental data modification
- Enrollment allows grade mutation for legitimate updates

### 2. Duplicate Prevention & Data Integrity
- TreeMap storage automatically maintains sorted order
- Each manager checks for existing IDs before adding
- DuplicateDataException prevents duplicate entries
- Important for maintaining data consistency in concurrent scenarios

### 3. Exception Handling Order
- EnrollmentManager validates COURSE first (per spec)
- Then validates STUDENT second
- Ensures consistent error reporting in exact order
- Critical for exact output matching in tests

### 4. Manager Pattern (Static)
- Not true singleton, but static methods provide same benefits
- Thread-safe through TreeMap's internal synchronization
- Provides centralized data access point
- Simplifies resource management

### 5. TreeMap for Sorting
- Automatically sorts by String keys (ID)
- No manual sorting needed in output
- O(log n) insertion and retrieval
- Provides iteration in sorted order

## Code Quality Standards

### Java Conventions
- Pascal case for class names: Course, Student, Enrollment
- Camel case for methods and variables: addCourse(), courseManager
- Private/static keywords for encapsulation
- Full documentation with JavaDoc comments

### Separation of Concerns
- Model classes: Data representation only
- Manager classes: Data access and validation
- Driver classes: User interface and orchestration

### Error Handling
- Exception-based error reporting (type-safe)
- Specific exceptions for specific errors
- No silent failures (logged or reported)

### No Hardcoding
- Used parameterized methods throughout
- Input parsing from command format
- Dynamic storage without pre-allocated sizes
- Configuration-independent implementation
